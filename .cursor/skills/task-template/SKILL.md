# Шаблон задачи jira-helper

## Стандарты

### Статусы

Используй только эти статусы (одинаковые в файле задачи и в EPIC таблице):

| Статус | Описание |
|--------|----------|
| `TODO` | Не начата |
| `IN_PROGRESS` | В работе |
| `VERIFICATION` | Агент завершил, ожидает ручной проверки пользователем |
| `DONE` | Завершена и проверена пользователем |

### Именование файлов

`TASK-{N}-{короткое-описание}.md`

Примеры:
- `TASK-20-add-tests-coverage.md`
- `TASK-6-create-settings-button-view.md`

### Нумерация

**Перед созданием задачи** определи следующий номер:

```bash
ls tasks/TASK-*.md | sed 's/.*TASK-\([0-9]*\).*/\1/' | sort -n | tail -1
```

Используй `max + 1`. Текущий диапазон: TASK-1..TASK-172.

### Standalone задачи

Если задача не входит в EPIC (миграция инструментов, одноразовый рефакторинг):

```markdown
**Parent**: Standalone
```

Standalone задача:
- Не добавляется в EPIC таблицу
- Имеет собственные критерии приёмки
- Нумеруется в общей последовательности

## Типы задач

### Bug Fix

Обязательные поля:
- Root cause с кодом
- Ссылка на .feature сценарий (если есть)

### Feature / Refactoring

Обязательные поля:
- Целевой код (до/после или шаблон)
- Ссылки на файлы-референсы

### Тестирование

Обязательные поля:
- Таблица покрытия (Scenario → Cypress Test)
- Команды запуска тестов в критериях приёмки

## Шаблон EPIC

`EPIC-{N}-{короткое-описание}.md`

```markdown
# EPIC-{N}: {Название}

**Status**: TODO
**Created**: {YYYY-MM-DD}

---

## Цель

{Что решаем и зачем. Проблема → Решение.}

## Target Design

См. [target-design-{name}.md](./target-design-{name}.md)

## Архитектура

{Mermaid-диаграмма — обзор изменений}

## Задачи

### Phase 1: {Название фазы}

| # | Task | Описание | Status |
|---|------|----------|--------|
| {N} | [TASK-{N}](./TASK-{N}-{name}.md) | {Описание} | TODO |

### Phase 2: {Название фазы}

| # | Task | Описание | Status |
|---|------|----------|--------|
| {N} | [TASK-{N}](./TASK-{N}-{name}.md) | {Описание} | TODO |

## Dependencies

{Mermaid flowchart — граф зависимостей между задачами}

**Параллельно можно выполнять:**
- {список независимых задач}

**Последовательно:**
- {цепочки зависимостей}

## Acceptance Criteria

- [ ] {Критерий 1}
- [ ] {Критерий 2}
- [ ] Все тесты проходят
- [ ] ESLint без ошибок
```

### Правила EPIC

- **Фазы** группируют задачи по логическим блокам (инфраструктура → модели → компоненты → cleanup)
- **Dependencies** — Mermaid граф показывает что можно запускать параллельно
- **Acceptance Criteria** — для EPIC в целом, не для отдельных задач
- **Target Design** — ссылка на документ из `solution-design` скилла (если есть)

---

## EPIC таблица

При создании задачи — добавь строку в EPIC таблицу:

```markdown
| {N} | [TASK-{N}](./TASK-{N}-{name}.md) | {Описание} | TODO |
```

При завершении — обнови статус в обоих местах (файл задачи + EPIC таблица).
