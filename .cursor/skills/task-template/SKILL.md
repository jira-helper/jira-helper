# Шаблон задачи jira-helper

## Папка фичи (feature bundle)

Всё, что относится к **одному** запросу/эпику, лежит в **одной** директории — её можно удалить целиком.

### Старт из чата (без ручной подготовки)

Пользователь **не обязан** заранее создавать папку или `request.md`. Достаточно описать фичу в чате (лучше с агентом **feature-orchestrator**).

**Агент в первом же ответе должен:**

1. Выбрать `<FEATURE-SLUG>` (kebab-case; при коллизии имён папок — уточнить у пользователя или предложить суффикс).
2. Создать `.agents/tasks/<FEATURE-SLUG>/` и файл **`request.md`** с датой, заголовком и текстом запроса из чата.
3. Дальше вести EPIC/TASK/requirements **только** в этой папке.

Если пользователь **уже** указал существующую папку (например «продолжи в `.agents/tasks/foo-bar/`») — не дублируй slug, используй её и дополни `request.md` новым сообщением или секцией «Дополнение».

**Базовый путь:**

```
.agents/tasks/<FEATURE-SLUG>/
```

**`<FEATURE-SLUG>`**: латиница, kebab-case, отражает тему (например `field-limits-modernization`, `shared-di-refactoring`). Не используй пробелы.

**Рекомендуемые файлы внутри папки:**

| Файл | Назначение |
|------|------------|
| `request.md` | Исходный запрос пользователя (копия или ссылка на контекст) |
| `requirements.md` | Согласованные требования и ограничения |
| `EPIC-{N}-{short-name}.md` | EPIC с таблицей задач |
| `target-design.md` или `target-design-{name}.md` | Target Design (по скиллу solution-design) |
| `*.feature` | BDD-сценарии — **если** хранишь их здесь; иначе ссылка в EPIC на путь в `src/` |
| `TASK-{N}-{short}.md` | Атомарные задачи |
| `delivery.md` | Итог релиза, changelog, ссылки (опционально) |

**Legacy:** файлы прямо в `.agents/tasks/*.md` (без подпапки) — старые эпики; **новые** артефакты создавай только внутри `.agents/tasks/<FEATURE-SLUG>/`.

**Одноразовая задача без полноценного EPIC:** подпапка `.agents/tasks/_standalone/<topic-slug>/` с `TASK-{N}-*.md` (и при необходимости `request.md`); **Parent**: Standalone.

## Стандарты

### Статусы

Используй только эти статусы (одинаковые в файле задачи и в EPIC таблице):

| Статус | Описание |
|--------|----------|
| `TODO` | Не начата |
| `IN_PROGRESS` | В работе |
| `VERIFICATION` | Агент завершил, ожидает ручной проверки пользователем |
| `DONE` | Завершена и проверена пользователем |

### Именование файлов

`TASK-{N}-{короткое-описание}.md`

Примеры:
- `TASK-20-add-tests-coverage.md`
- `TASK-6-create-settings-button-view.md`

### Нумерация

Номера **TASK** и **EPIC** глобальные по репозиторию (как раньше), независимо от папки фичи.

**Перед созданием задачи** определи следующий номер (все подпапки + legacy в корне):

```bash
find .agents/tasks -name 'TASK-*.md' -type f 2>/dev/null | sed 's/.*TASK-\([0-9]*\).*/\1/' | sort -n | tail -1
```

Используй `max + 1`.

**Перед созданием EPIC** — следующий номер EPIC:

```bash
find .agents/tasks -name 'EPIC-*.md' -type f 2>/dev/null | sed 's/.*EPIC-\([0-9]*\).*/\1/' | sort -n | tail -1
```

Используй `max + 1`.

## Шаблон TASK

```markdown
# TASK-{N}: {Краткое название}

**Status**: TODO

**Parent**: [EPIC-{N}](./EPIC-{N}-{name}.md)

## Описание

{Что нужно сделать и зачем. 2-3 предложения.}

## Файлы

{Список файлов для создания/изменения}

\`\`\`

├── file1.ts        # новый / изменение
└── file2.test.ts   # новый
\`\`\`

## Что сделать

{Пронумерованные шаги}

1. {Шаг 1}
2. {Шаг 2}
3. {Шаг 3}

## Критерии приёмки

- [ ] {Критерий 1}
- [ ] {Критерий 2}
- [ ] Тесты проходят: `npm test`
- [ ] Нет ошибок линтера: `npm run lint:eslint -- --fix`

## Зависимости

{Ссылки на другие задачи, файлы, референсы}

- Зависит от: [TASK-{N}](./TASK-{N}-{name}.md)
- Референс: `src/path/to/similar-implementation`

## Результаты

**Дата**: {YYYY-MM-DD}

**Агент**: {Coder / Architect / Manual}

**Статус**: VERIFICATION

**Что сделано**:

{Краткое описание выполненных изменений — 2-5 пунктов}

- Обновлён файл Z
- Добавлены тесты

**Проблемы и решения**:

{ОБЯЗАТЕЛЬНО! Документируй ВСЕ проблемы, с которыми столкнулся, и как их решил.}

```

### Standalone задачи

Если задача не входит в EPIC (миграция инструментов, одноразовый рефакторинг):

```markdown
**Parent**: Standalone
```

Standalone задача:
- Не добавляется в EPIC таблицу
- Имеет собственные критерии приёмки
- Нумеруется в общей последовательности

## Типы задач

### Bug Fix

Обязательные поля:
- Root cause с кодом
- Ссылка на .feature сценарий (если есть)

### Feature / Refactoring

Обязательные поля:
- Целевой код (до/после или шаблон)
- Ссылки на файлы-референсы

### Тестирование

Обязательные поля:
- Таблица покрытия (Scenario → Cypress Test)
- Команды запуска тестов в критериях приёмки

## Шаблон EPIC

`EPIC-{N}-{короткое-описание}.md`

```markdown
# EPIC-{N}: {Название}

**Status**: TODO
**Created**: {YYYY-MM-DD}

---

## Цель

{Что решаем и зачем. Проблема → Решение.}

## Target Design

См. [target-design-{name}.md](./target-design-{name}.md) (тот же каталог, что и EPIC — папка фичи)

## Архитектура

{Mermaid-диаграмма — обзор изменений}

## Задачи

### Phase 1: {Название фазы}

| # | Task | Описание | Status |
|---|------|----------|--------|
| {N} | [TASK-{N}](./TASK-{N}-{name}.md) | {Описание} | TODO |

### Phase 2: {Название фазы}

| # | Task | Описание | Status |
|---|------|----------|--------|
| {N} | [TASK-{N}](./TASK-{N}-{name}.md) | {Описание} | TODO |

## Dependencies

{Mermaid flowchart — граф зависимостей между задачами}

**Параллельно можно выполнять:**
- {список независимых задач}

**Последовательно:**
- {цепочки зависимостей}

## Acceptance Criteria

- [ ] {Критерий 1}
- [ ] {Критерий 2}
- [ ] Все тесты проходят
- [ ] ESLint без ошибок
```

### Правила EPIC

- **Фазы** группируют задачи по логическим блокам (инфраструктура → модели → компоненты → cleanup)
- **Dependencies** — Mermaid граф показывает что можно запускать параллельно
- **Acceptance Criteria** — для EPIC в целом, не для отдельных задач
- **Target Design** — ссылка на документ из `solution-design` скилла (если есть)

---

## EPIC таблица

При создании задачи — добавь строку в EPIC таблицу:

```markdown
| {N} | [TASK-{N}](./TASK-{N}-{name}.md) | {Описание} | TODO |
```

При завершении — обнови статус в обоих местах (файл задачи + EPIC таблица).
