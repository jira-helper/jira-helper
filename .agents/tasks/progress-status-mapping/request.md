# Request: Настроить маппинг статусов для прогресс-бара

**Дата**: 2026-04-28
**Slug**: `progress-status-mapping`
**Тип**: UI
**Has UI**: yes
**Scope**: large

## Описание запроса

Источник: GitHub issue #21 `Feature: Настроить маппинг статусов для прогресс-бара в jira-helper`.

Нужно добавить возможность настраивать, к какой нормализованной категории прогресса относится каждый Jira-статус, чтобы прогресс-бар корректно работал для команд с кастомными workflow.

Проблема:

- Сейчас прогресс-бар определяет категорию задачи (To Do / In Progress / Done) на основе категории статуса из Jira API.
- Задачи в статусе `Ready for Release` и аналогичных могут относиться к Jira-категории `To Do`.
- Из-за этого почти готовые задачи считаются не начатыми, прогресс эпика/свимлейна искажается.

Что нужно сделать:

- Добавить настройку маппинга `status -> нормализованная категория прогресса`.
- Пользователь должен уметь настроить маппинг под workflow команды.
- Конфигурация должна сохраняться между сессиями.
- По умолчанию должны работать стандартные Jira-статусы `To Do`, `In Progress`, `Done`.

Изначальные варианты из issue:

1. Настройка маппинга статусов.
2. Расширенные категории прогресса.
3. Авто-определение по workflow.

## Уточнения

### Из комментария к GitHub issue

- Настройку делаем в двух местах:
  - отдельная опция/настройка в ганте;
  - отдельная опция/настройка в фиче прогресса по задачам (progress-bar / sub-tasks progress).
- Маппим только связь `status -> нормализованная категория прогресса`.
- Без усложнений по issue type, типам связей и другим измерениям.
- Один маппинг по статусу, без комбинированных правил.

### Раунд 1

- Для первой реализации выбран вариант `status -> существующие buckets прогресса`, без кастомных новых сегментов и без авто-анализа Jira workflow.
- Пользователю доступны buckets: `To Do`, `In Progress`, `Done`.
- Хранение оставляем раздельным по текущим feature-паттернам:
  - sub-tasks progress использует Jira board property;
  - gantt использует текущий механизм localStorage.
- Сопоставление статуса должно использовать `status id` как единственный ключ. Сохранённое `status name` допустимо только как fallback/debug label, если актуальное имя из Jira statuses ещё не загружено или статус недоступен.
- UI должен поддерживать автокомплит из существующих статусов. Если статус не найден в подсказках, произвольный текст не сохраняется как статус.
