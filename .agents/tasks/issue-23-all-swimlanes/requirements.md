# Requirements: Issue #23 - All swimlanes behavior

**Feature folder**: `.agents/tasks/issue-23-all-swimlanes/`
**Связанный request**: [request.md](./request.md)
**Дата**: 2026-04-27
**Статус**: draft
**Has UI**: yes

## 1. Цель и мотивация

Сделать поведение выбора свимлайнов в настройках лимитов однозначным. Пользователь должен видеть либо режим "все свимлайны", либо режим ручного выбора, но не оба способа задания одного и того же состояния одновременно.

Критерий успеха: при включенном **All swimlanes** список свимлайнов скрыт, а при снятом **All swimlanes** список показан и пользователь явно выбирает нужные свимлайны.

## 2. Пользователи и контекст

- Роли / контекст использования: пользователь, настраивающий group limits или personal WIP limits на Jira board settings.
- Страницы Jira и точки входа: экран настроек расширения, блок **Swimlanes** внутри форм лимитов.
- Основной shared-компонент: `src/shared/components/SwimlaneSelector/SwimlaneSelector.tsx`.
- Текущие места использования: `ColumnLimitsForm` и `PersonalWipLimitContainer`.

## 3. Функциональные требования

1. **FR-1**: Если `value` означает "all" (`[]` по текущей convention), компонент показывает checked **All swimlanes** и не показывает список отдельных свимлайнов.
2. **FR-2**: Если пользователь снимает **All swimlanes**, компонент показывает список всех доступных свимлайнов, а сам чекбокс **All swimlanes** становится unchecked.
3. **FR-3**: В режиме ручного выбора список содержит все `swimlanes`, переданные в props, с индивидуальными checkbox.
4. **FR-4**: Если пользователь снова включает **All swimlanes**, список скрывается и наружу отправляется `onChange([])`.
5. **FR-5**: Если пользователь выбирает все индивидуальные свимлайны вручную, состояние нормализуется в `[]` и UI возвращается к режиму "all".
6. **FR-6**: Поведение должно сохранять существующую data convention: `[]` означает "all swimlanes"; непустой массив означает ручной выбор конкретных ids.
7. **FR-7**: Изменение должно работать одинаково в `ColumnLimitsForm` и `PersonalWipLimitContainer` без добавления локальной бизнес-логики в эти формы.

## 4. Сценарии

### S1: Режим "все свимлайны"

- Given доступно несколько свимлайнов
- And текущее значение равно `[]`
- When пользователь смотрит блок Swimlanes
- Then checkbox **All swimlanes** checked
- And список отдельных свимлайнов не показан

### S2: Переход к ручному выбору

- Given checkbox **All swimlanes** checked
- When пользователь снимает checkbox **All swimlanes**
- Then checkbox **All swimlanes** становится unchecked
- And появляется список всех доступных свимлайнов

### S3: Возврат в режим "все"

- Given пользователь находится в режиме ручного выбора
- When пользователь включает checkbox **All swimlanes**
- Then список свимлайнов скрывается
- And `onChange` получает `[]`

### S4: Нормализация полного ручного выбора

- Given пользователь находится в режиме ручного выбора
- When пользователь отмечает все доступные свимлайны
- Then `onChange` получает `[]`
- And UI возвращается к режиму "all": checked **All swimlanes**, список скрыт

## 5. Данные и миграции

- Источник истины данных: существующие формы лимитов и их persisted settings.
- Миграции / совместимость: миграция данных не нужна. Convention `[] = all swimlanes` сохраняется.
- Обратная совместимость: существующие сохраненные настройки с `[]` продолжают означать "все".

## 6. Нефункциональные требования

- Тестирование: добавить/обновить Cypress component tests для `SwimlaneSelector`; обязательно проверить реальный клик по **All swimlanes** и видимость/скрытие списка в браузере.
- Дополнительное покрытие UI: добавить либо Storybook `play` interaction для переключения all/manual, либо screenshot/visual проверку, если в проекте уже есть готовый паттерн для этого компонента.
- Accessibility: сохранить нативные checkbox controls Ant Design, не ломать label-click behavior.
- Performance: не добавлять запросы к Jira и не менять формат persisted data.

## 7. Вне scope

- Не менять модель хранения лимитов.
- Не менять список доступных свимлайнов и стратегию получения swimlanes из Jira.
- Не добавлять новый дизайн-системный компонент вместо текущего `SwimlaneSelector`.
- Не менять поведение выбора колонок и issue types.

## 8. Открытые вопросы

- [ ] Нужно ли при снятии **All swimlanes** сразу выбирать все individual checkbox визуально или начинать с пустого ручного выбора? Рекомендуемое решение по issue: показать список для явного выбора без отправки `onChange` до выбора конкретных пунктов.

## 9. Черновик критериев приёмки

- [ ] При `value=[]` виден только checked **All swimlanes**, список скрыт.
- [ ] При снятии **All swimlanes** список появляется, а checkbox становится unchecked.
- [ ] При повторном включении **All swimlanes** список скрывается и сохраняется `[]`.
- [ ] При ручном выборе всех пунктов состояние нормализуется в `[]` и список скрывается.
- [ ] Cypress component tests покрывают основные переходы.
- [ ] Есть дополнительная UI-проверка: Storybook `play` interaction или visual/screenshot test по существующему проектному паттерну.

## 10. UI Wireframe

### Режим "все свимлайны"

```
+-- Swimlanes -------------------------+
| [x] All swimlanes                    |
+--------------------------------------+
```

**Легенда:**
- `[x]` — checked checkbox.
- Список individual swimlanes в этом режиме отсутствует.

### Режим ручного выбора

```
+-- Swimlanes -------------------------+
| [ ] All swimlanes                    |
|                                      |
| +-- available swimlanes ------------+ |
| | [x] Frontend                      | |
| | [ ] Backend                       | |
| | [x] Expedite                      | |
| +-----------------------------------+ |
+--------------------------------------+
```

**Легенда:**
- `[ ] All swimlanes` — unchecked, включен ручной режим.
- Список содержит все доступные swimlanes.
