# Требования (requirements) jira-helper

## Назначение

Превратить запрос пользователя в **один согласованный документ** `requirements.md` в папке фичи `.agents/tasks/<FEATURE-SLUG>/`, чтобы архитектор и разбиение на TASK опирались на явный текст, а не на разрозненный чат.

## Вход

- `.agents/tasks/<FEATURE-SLUG>/request.md` — исходное сообщение/контекст (создаётся на этапе 0 оркестратора), включая секцию «Уточнения» с ответами из interview.
- Метаданные из `request.md`: **Тип** (UI / logic / refactoring / bugfix), **Has UI** (yes / no), **Scope**.
- При необходимости: ссылки на issue, скриншоты в чате, похожие фичи в коде.

**Опционально** просмотри `docs/architecture_guideline.md`, чтобы вопросы про UI/state не противоречили принятым паттернам (не проектируй решение — только уточняй границы).

## Выход

Файл **`requirements.md`** в **той же папке**, что и `request.md`.

После сохранения кратко перечисли пользователю: цель, ключевые сценарии, явные ограничения и что осталось открытым (если есть).

## Чек-лист вопросов (задай то, что ещё не ясно из request)

### 1. Контекст и цель

- Какая **проблема** или **возможность**? Для кого (роль на доске / в настройках)?
- **Критерий успеха** одной фразой: что должно стать возможным или исчезнуть?
- Это **новая фича**, **изменение поведения** или **рефакторинг** без смены UX?

### 2. Место в продукте (jira-helper)

- Где в UI: **какая страница Jira** (доска, бэклог, issue view, настройки расширения и т.д.)?
- Точка входа: **кнопка**, **меню**, **автоматическое поведение**, **настройка в модалке**?
- Затрагиваются ли **несколько board / project** — нужна ли изоляция настроек по контексту?

### 3. Поведение и сценарии

- **Основной сценарий** (happy path) — шаги пользователя.
- **2–4 альтернативы или края**: пустые данные, отмена, нет прав, медленный ответ Jira, конфликт с другой фичей расширения.
- **Состояние до/после**: что сохраняется (board property, local storage, только сессия)?

### 4. Данные и совместимость

- Новые поля в **существующем property** или **новый ключ**? Нужна ли **миграция** старых данных?
- Обратная совместимость: старые версии расширения / старые сохранённые данные?
- Ограничения по **производительности** (частота вызовов API, размер DOM)?

### 5. UI и визуальное поведение (если `has_ui: yes`)

- **Layout**: где размещается компонент на странице? Рядом с чем? Над/под чем?
- **Элементы управления**: какие элементы (кнопки, инпуты, дропдауны, таблицы, модалки)?
- **Состояния**: как выглядит при пустых данных? При загрузке? При ошибке?
- **Интерактивность**: drag-n-drop, inline editing, hover-эффекты, анимации?
- **Адаптивность**: фиксированная ширина или растягивается? Есть ли скролл?

### 6. Нефункциональное

- **Доступность**: фокус, клавиатура, контраст — есть ли ожидания?
- **Тестирование**: что критично покрыть **Cypress** (DnD, визуал) vs **Vitest** (модели)?

### 7. Границы

- **Вне scope**: что явно не делаем в этой итерации?
- Зависимости от **внешних** факторов (версия Jira, флаг, другая задача)?

## Правила

1. **Не проектируй архитектуру** в этом документе — только требования и ограничения; Target Design — следующий этап.
2. Не придумывай ответы: явно помечай **TBD** и вынеси в «Открытые вопросы».
3. Используй **всю информацию** из `request.md`, включая секцию «Уточнения» с ответами из interview (этап 1a).
4. Когда пользователь подтвердил содержание — выставь **Статус: agreed** (или оставь draft, если ждёте решений по TBD).

## Правила ASCII Wireframe (для UI-фич)

ASCII wireframes добавляются в секцию `## 10. UI Wireframe` на этапе 1d оркестратором. Эти правила описывают формат.

### Когда нужен wireframe

- `Has UI: yes` в `request.md` / `requirements.md`
- Фича создаёт или существенно меняет визуальные элементы

### Что визуализировать

1. **Основной экран** (happy path) — как выглядит результат
2. **Модалки / панели** — если есть интерактивные формы
3. **Edge-состояния** (по необходимости): пустые данные, загрузка, ошибка

### Синтаксис элементов

```
Контейнеры:    +--- Заголовок ---+    или    +--[ Tab Name ]--+
               |  содержимое     |           |                |
               +-----------------+           +----------------+

Кнопки:        [Save]  [Cancel]  [+ Add item]

Dropdown:      [Option selected v]

Checkbox:      [x] Enabled    [ ] Disabled

Radio:         (o) Selected   ( ) Not selected

Input:         [_____________]    или    [placeholder text___]

Таблица:       | Column A | Column B | Column C |
               |----------|----------|----------|
               | value    | value    | value    |

Текст:         просто текст

Аватар:        @username

Иконка:        (i) info    (!) warning    (*) star

Разделитель:   ────────────────────
```

### Правила оформления

- Каждый wireframe — в блоке ` ``` ` (code block без языка) для моноширинности
- После wireframe — **легенда** с пояснением нестандартных обозначений
- Несколько экранов = отдельные подсекции (`### Название экрана`)
- Показывай **структуру и иерархию**, не пиксельную точность
- Используй реалистичные данные вместо placeholder (имена, числа, статусы)
- Ширина — не более 60 символов (читаемость в markdown preview)

### Пример

```
### Панель настроек Person Limits

+--[ Person Limits ]---------------------------+
|                                               |
|  Person         | Limit | Swimlane            |
|  ---------------|-------|---------------------|
|  @john.doe      |   5   | [All           v]   |
|  @jane.smith    |   3   | [Backend       v]   |
|                                               |
|  [+ Add person]                               |
|                                               |
|           [Cancel]            [Save]          |
+-----------------------------------------------+

Легенда:
  @user       — аватар + имя пользователя
  [... v]     — dropdown со списком свимлейнов
  [+ Add ...] — кнопка добавления строки
```

## Связь с оркестратором

- **Этап 1a** (interview): оркестратор использует чек-лист вопросов из этого skill для формулировки вопросов пользователю.
- **Этап 1b** (draft): саб-агент читает этот skill и генерирует `requirements.md`.
- **Этап 1d** (wireframe): оркестратор использует правила ASCII Wireframe из этого skill.
- **Этап 2** (architect): использует **agreed** `requirements.md` (включая wireframes) как основной вход.
