# QA Check

## Назначение

Автоматическая и полуавтоматическая проверка кода задачи: сборка, тесты, линтер + проектные требования. Результат фиксируется в файле-отчёте.

## Вход

- **TASK файл** — какие файлы затронуты.
- Код в рабочем дереве (после coder или после фикса).

## Выход

Файл **`QA-TASK-{N}.md`** в **той же папке фичи** (`.agents/tasks/<FEATURE-SLUG>/`).

---

## Проверки

### 1. Автоматические (обязательные)

Запусти **последовательно**, каждую команду отдельно:

```bash
npm run lint:eslint -- --fix
npm test
npm run build:dev
```

Зафиксируй результат каждой: **pass** или **fail** + текст ошибки.

### 2. Проектные требования (проверь вручную по коду)

#### i18n

- Все пользовательские строки (labels, tooltips, сообщения) используют ключи из `texts` (или соответствующего i18n-механизма проекта)?
- Нет захардкоженных русских/английских строк в компонентах и моделях?
- Если добавлены новые ключи — они есть для **обоих** языков?

#### Accessibility (базовый уровень)

- Интерактивные элементы имеют `aria-label` или видимый текст?
- Фокус не теряется при взаимодействии (модалки, dropdown)?

#### Storybook

- Если TASK создаёт View-компоненты — есть ли Storybook stories?
- Stories рендерятся без ошибок?

### 3. Регрессия (если scope большой)

- Другие фичи расширения не сломаны (визуально проверь в Storybook или по тестам)?

## Формат отчёта `QA-TASK-{N}.md`

```markdown
# QA: TASK-{N} — {название}

**Дата**: YYYY-MM-DD
**TASK**: [TASK-{N}](./TASK-{N}-*.md)
**Вердикт**: PASS | FAIL

## Автоматические проверки

| Проверка | Результат | Детали |
|----------|-----------|--------|
| ESLint | pass/fail | {ошибки если есть} |
| Tests | pass/fail | {failed tests если есть} |
| Build | pass/fail | {ошибки если есть} |

## Проектные требования

| Проверка | Результат | Комментарий |
|----------|-----------|-------------|
| i18n | pass/fail | {детали} |
| Accessibility | pass/fail | {детали} |
| Storybook | pass/fail/N/A | {детали} |

## Проблемы

{Если вердикт FAIL — список проблем с указанием файлов.}

## Резюме

{1–2 предложения.}
```

---

## Правила

1. Вердикт **FAIL** — если хотя бы одна автоматическая проверка fail **или** i18n fail.
2. Вердикт **PASS** — всё зелёное.
3. **Не исправляй код** — только фиксируй проблемы в отчёте.
4. При lint `--fix` автоматически применяет некоторые правки — это нормально, но если после fix остаются ошибки — это fail.
